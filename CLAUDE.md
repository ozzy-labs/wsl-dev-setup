# Claude向けリポジトリガイド

このドキュメントは、Claude（AI）がこのリポジトリを理解し、適切にサポートするための情報です。

## リポジトリの目的

このリポジトリは、WSL2/Ubuntu環境での開発環境を自動的にセットアップするためのスクリプト集です。開発に必要なツールやソフトウェアを一括でインストール・設定することで、環境構築の手間を削減します。

## 重要な原則

1. **公式推奨に従う**: ソフトウェアのインストールは公式ドキュメントの推奨方法に従う
2. **冪等性の確保**: スクリプトを複数回実行しても安全に動作すること
3. **エラーハンドリング**: 失敗時に適切なメッセージと対処方法を表示
4. **日本語優先**: このリポジトリは日本語で記録される

## ファイル構成

### セットアップスクリプト

- `install.sh`: GitHub からのワンライナー実行とローカル実行の両方に対応するエントリーポイント
- `scripts/setup-local-ubuntu.sh`: ローカル開発環境セットアップスクリプト（メイン）
- `scripts/setup-zsh-ubuntu.sh`: Zsh環境セットアップスクリプト

### ドキュメント

- `README.md`: プロジェクト概要（英語）
- `README.ja.md`: プロジェクト概要（日本語）

### 設定ファイル

- `.github/`: GitHub関連の設定（IssueテンプレートなしPR テンプレート、ワークフロー）
- `.claude/`: Claude Code用の設定やコマンド
- `docs/`: 追加ドキュメント

## Git操作のルール

**【重要】すべてのGit操作はユーザーの明示的な承認が必要です**

### 禁止事項
- ユーザーの承認なしに `git add`, `git commit`, `git push`, `git pull`, `git merge` 等のGit操作を実行しない
- 自動的にコミットやプッシュを行わない
- Git操作を提案する際は、必ず実行前にユーザーに確認を求める

### 許可される行動
- Git操作のコマンドを提案する（実行はしない）
- `git status`, `git log`, `git diff` などの読み取り専用操作を提案する
- ユーザーが明示的に「Git操作を実行して」と指示した場合のみ実行

### Gitワークフローの実行タイミング
- **機能ブランチ作成、コミット、プッシュ、PR作成、PRマージの一連のGitワークフローは、ユーザーから明示的に指示があった場合のみ実行する**
- ファイルの作成や変更のみを依頼された場合は、Gitワークフローを実行せずに待機する
- 例:
  - **❌ NG**: 「README.md を追加して」→ ファイル作成後、勝手にブランチ作成・コミット・PR作成
  - **✅ OK**: 「README.md を追加して」→ ファイル作成のみ、Gitワークフローは実行せず
  - **✅ OK**: 「README.md を追加して。その後、機能ブランチ作成からPRマージまで実行して」→ 全ワークフロー実行

### コミットメッセージとPRの言語・形式

#### Conventional Commitsルール（必須）

このリポジトリでは **Conventional Commits** のルールに従います。ただし、**日本語で記述**します。

**【重要】すべてのコミットメッセージとPRタイトルには `type:` prefixが必須です**

**基本フォーマット**:
```
<type>: <subject>

<body>（任意）

<footer>（任意）
```

**Type（種類）**:
- `feat`: 新機能の追加（**注**: `feature` ではなく `feat` を使用）
- `fix`: バグ修正
- `docs`: ドキュメントのみの変更
- `style`: コードの動作に影響しない変更（フォーマット、空白など）
- `refactor`: バグ修正や機能追加を伴わないコードの改善
- `perf`: パフォーマンス改善
- `test`: テストの追加や修正
- `chore`: ビルドプロセスやツールの変更
- `ci`: CI設定ファイルやスクリプトの変更

**注意**: typeは省略形を使用（`feat`, `perf`, `refactor` など）。`feature`, `performance` などのフルスペルは使用しない

**日本語での記述ルール**:
- **type**: 英語（Conventional Commits標準）
- **subject**: 日本語で簡潔に（50文字以内推奨）
- **body**: 日本語で詳細説明（任意）
- **footer**: 関連IssueやBreaking Changesを記載（任意）

**例**:
```
feat: Docker公式リポジトリからのインストールに対応

docker.ioパッケージではなく、Docker公式リポジトリから
docker-ce、docker-compose-pluginをインストールするように変更。

Closes #1
```

```
docs: README.jaに使用方法を追加

初心者向けにスクリプトの実行手順を詳細化。
```

```
fix: Zshインストール時のエラーハンドリングを改善
```

**PRのタイトルと本文**:
- **PRタイトル**: **必ず** `type: subject` フォーマットを使用（例: `feat: Docker公式インストールに対応`）
- **PR本文**: 日本語で変更内容を説明
- **関連Issue**: `Closes #1` などで明記

**重要**: PRタイトルにも必ず `feat:`, `fix:`, `docs:` などのtype prefixを付けること

**Issueのタイトルと本文**:
- **Issueタイトル**: type prefixは**不要**（例: `Docker公式リポジトリからのインストールに対応`）
- **Issue本文**: 日本語で実装内容や目的を説明
- Issueは「何をするか」の提案・要求なので、Conventional Commitsのtype prefixは付けない
- CommitとPRは「何をしたか」の記録なので、type prefixが必要

#### NGとOKの例
- **❌ NG**: 「変更をコミットしました」と言って勝手にコミット
- **✅ OK**: 「以下のコマンドでコミットできます：`git add . && git commit -m "..."`」と提案
- **✅ OK**: ユーザーが「コミットして」と明示的に指示した場合のみ実行
- **❌ NG**: 英語のみのコミットメッセージ「feat: initial setup」
- **❌ NG**: typeなしの日本語「初回セットアップ完了」
- **✅ OK**: Conventional Commits + 日本語「feat: 初回セットアップ完了」

## Claudeのサポート方針

### スクリプト修正時
- **DO**: 公式ドキュメントの推奨方法を確認する
- **DO**: 既存のコードスタイルとエラーハンドリングパターンに従う
- **DO**: 変更前にスクリプト全体を読み、影響範囲を理解する
- **DON'T**: 非公式な方法やレガシーな手法を使わない

### 提案とインタラクション
- **DO**: 複数の実装方法がある場合、すべての選択肢を提示し、推奨案を明示する
- **DO**: 推奨理由を明確に説明する（保守性、拡張性、ベストプラクティス等）
- **DON'T**: 推奨案を押し付けず、最終判断はユーザーに委ねる

### ドキュメント作成時
- **DO**: 簡潔で分かりやすく
- **DO**: コード例を含める
- **DON'T**: 過度に詳細にしすぎない
