# レビューチェックリスト

このドキュメントは、Pull Request レビュー時に使用する詳細なチェックリストです。

[レビュープロセスガイド](review-process.md) と合わせて活用してください。

---

## Phase 1: 変更内容の詳細レビュー

**スコープ**: 変更ファイル自体

### 正確性

- [ ] 構文エラーがない
- [ ] スタイル違反がない（Markdown lint、YAML lint等）
- [ ] ロジックが正しく実装されている
- [ ] Issue の要件をすべて満たしている
- [ ] ファイル名が適切（命名規則に従っている）
- [ ] ファイル構造が適切（セクション分け、階層等）

### ベストプラクティス

- [ ] 命名規則に従っている
  - [ ] ファイル名: ケバブケース（例: `review-process.md`）
  - [ ] YAML ファイル: `.yaml` 拡張子
  - [ ] ディレクトリ名: 小文字、ハイフン区切り
- [ ] コード/ドキュメントの構造が適切
  - [ ] 見出しレベルが正しい（Markdown）
  - [ ] セクション分けが論理的
  - [ ] 適切な粒度で記述されている
- [ ] 可読性が高い
  - [ ] 適切なコメントや説明がある
  - [ ] わかりやすい記述（専門用語の説明等）
  - [ ] 例示が適切に含まれている
- [ ] DRY原則に従っている（重複がない）

### セキュリティ

- [ ] 機密情報が含まれていない
  - [ ] パスワード
  - [ ] APIキー、トークン
  - [ ] 個人情報（氏名、住所、メールアドレス等）
  - [ ] 秘密鍵
- [ ] 不適切な権限設定がない
- [ ] 外部入力の検証が適切（該当する場合）

### 保守性

- [ ] 将来の変更に対応しやすい設計
- [ ] 適切なドキュメント・コメントがある
- [ ] モジュール化されている（該当する場合）
- [ ] テストが容易な構造（該当する場合）

---

## Phase 2: 影響範囲のレビュー

**スコープ**: 既存ファイル・関連ファイル

### 整合性

- [ ] 既存ドキュメントと矛盾していない
  - [ ] README.md
  - [ ] CLAUDE.md
  - [ ] docs/ 配下の他のドキュメント
- [ ] 既存コード（スクリプト、テンプレート等）と矛盾していない
- [ ] 依存関係が正しく保たれている
- [ ] リンクや参照が正しく機能する
  - [ ] 内部リンク（相対パス）
  - [ ] 外部リンク
  - [ ] アンカーリンク

### ベストプラクティス

- [ ] 既存の設計パターンに従っている
- [ ] 既存の記述スタイルに従っている
- [ ] 重複がない（既存ドキュメントと内容が重複していない）
- [ ] 不整合がない（既存の情報と矛盾がない）
- [ ] 既存のルールやガイドラインに準拠している

### セキュリティ

- [ ] 既存のセキュリティ設定を破壊していない
- [ ] 新たな脆弱性を導入していない
- [ ] `.gitignore` 等の設定が適切に保たれている
- [ ] 権限設定（Claude Code の permissions）が適切

### 影響評価

- [ ] 関連ドキュメントの更新が必要か確認
  - [ ] README.md: ドキュメント一覧の更新
  - [ ] CLAUDE.md: ルールやリンクの追加
  - [ ] CHANGELOG.md: 変更履歴の記録
- [ ] 他の機能やワークフローへの影響がないか確認
  - [ ] プロジェクトの他の機能やワークフロー
- [ ] 既存のテンプレートやコマンドが正しく動作するか

---

## Phase 3: プロジェクト全体の整合性レビュー

**スコープ**: プロジェクト全体

### 方針整合性

- [ ] プロジェクトの原則や方針に沿っている（CLAUDE.md 参照）
  - [ ] 公式推奨に従う
  - [ ] 冒等性の確保
  - [ ] エラーハンドリング
  - [ ] 日本語優先
- [ ] ディレクトリ構造やファイル配置が適切
  - [ ] 適切なディレクトリに配置されている
  - [ ] ファイル名がディレクトリの目的と一致している
- [ ] プロジェクトの目的や価値観と一致している

### ベストプラクティス

- [ ] 命名規則がプロジェクト全体で統一されている
  - [ ] ファイル名
  - [ ] ディレクトリ名
  - [ ] セクション名
- [ ] ドキュメント体系が一貫している
  - [ ] 見出しレベルの使い方
  - [ ] セクション構成のパターン
  - [ ] 用語の統一
- [ ] コーディングスタイルやフォーマットが統一されている
  - [ ] Markdown スタイル
  - [ ] YAML スタイル
  - [ ] インデント（2スペース、4スペース等）

### セキュリティ

- [ ] プロジェクト全体のセキュリティポリシーに準拠している
- [ ] プライバシー保護の観点で問題ない
  - [ ] 個人的な内容が慎重に扱われている
  - [ ] センシティブな情報が含まれていない
- [ ] 公開リポジトリとして適切
  - [ ] 外部に公開して問題ない内容
  - [ ] ライセンス表記が適切

### 拡張性

- [ ] 将来の拡張に対応できる設計
  - [ ] 新しいセクションやドキュメントを追加しやすい
  - [ ] 柔軟な構造になっている
- [ ] スケーラビリティの問題がない
  - [ ] ファイルが大きくなりすぎていない
  - [ ] 階層が深くなりすぎていない
- [ ] 他の機能との統合が容易
  - [ ] 独立性が保たれている
  - [ ] インターフェースが明確

## 最終チェック: マージ前の総合確認

**スコープ**: PR全体とGit状態

### 完全性

- [ ] すべてのレビューフェーズ（Phase 1, 2, 3）が完了している
- [ ] 発見された問題がすべて解決されている
- [ ] レビューコメントに未対応事項がない
- [ ] TODO コメントが残っていない（意図的に残す場合を除く）

### 品質保証

- [ ] PR テンプレートのテストプラン項目がすべて完了している
  - [ ] スクリプトの動作確認（該当する場合）
  - [ ] ドキュメントの整合性確認
  - [ ] 既存機能への影響確認
  - [ ] その他のテスト項目
- [ ] ドキュメントが最新
  - [ ] CHANGELOG.md に変更が記録されている
  - [ ] README.md が更新されている（必要な場合）
  - [ ] 関連ドキュメントが更新されている
- [ ] コミットメッセージが適切
  - [ ] Conventional Commits に従っている（`type: subject`）
  - [ ] 日本語で記述されている
  - [ ] 変更内容を正確に表している

### Git状態

- [ ] コミットメッセージが Conventional Commits に従っている
  - [ ] type が正しい（feat, fix, docs, style, refactor, test, chore等）
  - [ ] subject が簡潔で明確
  - [ ] body が適切（必要な場合）
- [ ] ブランチが最新の main から分岐している
  - [ ] `git log --oneline` でブランチポイントを確認
  - [ ] 必要に応じて `git rebase main` で更新
- [ ] コンフリクトがない
  - [ ] `git status` で確認
  - [ ] マージ可能な状態
- [ ] 不要なファイルがコミットされていない
  - [ ] `.DS_Store`, `Thumbs.db` 等のOSファイル
  - [ ] エディタの一時ファイル
  - [ ] ログファイル

### マージ可否判断

- [ ] すべてのチェックが通っている
- [ ] 承認可能な状態
- [ ] 追加修正が不要
- [ ] マージ後の影響は許容範囲内
- [ ] Issue のクローズ条件を満たしている

## 変更種別ごとの追加チェック

### feat（新機能）

#### Phase 1

- [ ] 設計が妥当
  - [ ] 機能の範囲が適切
  - [ ] 実装方法が適切
  - [ ] 必要十分な機能が含まれている
- [ ] ユーザビリティが高い
  - [ ] 使いやすいインターフェース
  - [ ] わかりやすいドキュメント

#### Phase 2

- [ ] 既存機能との統合が適切
  - [ ] 既存機能を破壊していない
  - [ ] 既存機能と自然に統合されている
- [ ] 関連ドキュメントが更新されている
  - [ ] README.md のコマンド一覧

#### Phase 3

- [ ] プロジェクト方針と整合している
  - [ ] プロジェクトの目的に貢献している
  - [ ] 将来的なビジョンと一致している

---

### fix（バグ修正）

#### Phase 1

- [ ] 根本原因が特定されている
  - [ ] バグの原因が明確
  - [ ] 再発防止策が含まれている
- [ ] 修正が正確
  - [ ] バグが確実に修正されている
  - [ ] エッジケースも考慮されている

#### Phase 2

- [ ] 副作用がない
  - [ ] 他の機能に影響していない
  - [ ] 新たなバグを導入していない
- [ ] 関連箇所も確認されている
  - [ ] 同様の問題が他にないか確認

#### Phase 3

- [ ] 同様のバグが他にないか確認
  - [ ] プロジェクト全体で類似パターンを検索
  - [ ] 再発防止のためのドキュメント更新

---

### docs（ドキュメント）

#### Phase 1

- [ ] 内容が正確
  - [ ] 技術的に正しい
  - [ ] 最新の情報に基づいている
- [ ] 可読性が高い
  - [ ] わかりやすい表現
  - [ ] 適切な例示
  - [ ] 論理的な構成

#### Phase 2

- [ ] 他のドキュメントとの整合性
  - [ ] 矛盾がない
  - [ ] 重複が適切に管理されている
  - [ ] リンクが正しく機能する

#### Phase 3

- [ ] ドキュメント体系全体での一貫性
  - [ ] 用語が統一されている
  - [ ] スタイルが統一されている
  - [ ] 全体の構造と整合している

---

### refactor（リファクタリング）

#### Phase 1

- [ ] 動作が変わっていない
  - [ ] 機能的に同一
  - [ ] テストが通る（該当する場合）

#### Phase 2

- [ ] 依存関係が保たれている
  - [ ] 既存機能への影響なし
  - [ ] インターフェースが維持されている

#### Phase 3

- [ ] 設計改善がプロジェクト全体に良い影響
  - [ ] コードの品質向上
  - [ ] 保守性の向上
  - [ ] 拡張性の向上

---

### style（スタイル修正）

#### Phase 1

- [ ] フォーマットが統一されている
  - [ ] インデント
  - [ ] 空白・改行
  - [ ] 記号の使い方

#### Phase 2

- [ ] 既存のスタイルとの一貫性
  - [ ] 他のファイルと同じスタイル
  - [ ] 一貫したフォーマット

#### Phase 3

- [ ] プロジェクト全体のスタイルガイドラインとの整合性
  - [ ] CLAUDE.md のルールに準拠
  - [ ] プロジェクトの慣例に従っている

---

### test（テスト追加）

#### Phase 1

- [ ] テストケースが適切
  - [ ] 網羅性が高い
  - [ ] 正確なテスト
  - [ ] エッジケースを含む

#### Phase 2

- [ ] 既存のテストと重複していない
  - [ ] 新しい観点のテスト
  - [ ] 既存テストを補完している

#### Phase 3

- [ ] テスト戦略全体との整合性
  - [ ] プロジェクトのテスト方針に従っている
  - [ ] 適切なテストレベル

#### Phase 1

- [ ] 設定が正しい
  - [ ] 構文が正しい

#### Phase 3

- [ ] プロジェクトのビルド・デプロイ戦略との整合性
  - [ ] 全体の戦略と一致
  - [ ] 将来の拡張を考慮

## チェックリスト使用のヒント

### 効率的な使い方

1. **すべての項目をチェックする必要はない**: 変更内容に応じて関連する項目のみチェック
2. **段階的にチェック**: Phase 1 → Phase 2 → Phase 3 → 最終チェックの順に進める
3. **優先順位をつける**: 重要な項目から確認する
4. **自動化を活用**: linter や formatter で自動チェックできる項目は自動化

### カスタマイズ

- プロジェクトの特性に応じて項目を追加・削除してOK
- 特定の種別の変更に特化したチェックリストを作成してもOK
- チェックリストは継続的に改善
