# レビュープロセスガイド

このドキュメントは、このプロジェクトにおける Pull Request のレビュープロセスを定義します。

## レビューの目的

1. **品質保証**: バグや不整合を事前に発見し、品質を担保する
2. **知識共有**: 変更内容をチームで共有し、理解を深める
3. **標準化**: プロジェクトの一貫性とベストプラクティスを維持する
4. **学習**: レビューを通じて改善点を発見し、継続的に学ぶ

## レビューのフロー

```
セルフレビュー → ピアレビュー → 最終チェック → マージ
```

1. **セルフレビュー**: 実装者（Claude）が自分の変更をレビュー
2. **ピアレビュー**: ユーザーが変更をレビュー（必要に応じて）
3. **最終チェック**: マージ前の総合確認
4. **マージ**: main ブランチへマージ

## レビュープロセス（3段階＋最終チェック）

### 基本構造

各フェーズは **スコープ（範囲） × 横断的観点（品質軸）** で構成されます。

| フェーズ | スコープ | 横断的観点 |
|---------|---------|-----------|
| Phase 1 | 変更ファイル自体 | 正確性、ベストプラクティス、セキュリティ、保守性 |
| Phase 2 | 既存・関連ファイル | 整合性、ベストプラクティス、セキュリティ、影響評価 |
| Phase 3 | プロジェクト全体 | 方針整合性、ベストプラクティス、セキュリティ、拡張性 |
| 最終チェック | PR全体とGit状態 | 完全性、品質保証、Git状態、マージ可否判断 |

---

### Phase 1: 変更内容の詳細レビュー

**スコープ**: 変更ファイル自体

#### 観点

**1. 正確性**:

- 構文エラーやスタイル違反がないか
- ロジックが正しく実装されているか
- Issue の要件を満たしているか
- ファイル名や構造が適切か

**2. ベストプラクティス**:

- 命名規則に従っているか（ファイル名、変数名、関数名等）
- コード/ドキュメントの構造が適切か
- 可読性が高いか（適切なコメント、わかりやすい記述）
- DRY原則（Don't Repeat Yourself）に従っているか

**3. セキュリティ**:

- 機密情報（パスワード、APIキー、個人情報等）が含まれていないか
- 不適切な権限設定がないか
- 外部入力の検証が適切か（該当する場合）

**4. 保守性**:

- 将来の変更に対応しやすい設計か
- 適切なドキュメント・コメントがあるか
- テストが容易な構造か（該当する場合）

---

### Phase 2: 影響範囲のレビュー

**スコープ**: 既存ファイル・関連ファイル

#### 観点

**1. 整合性**:

- 既存のドキュメント・コードと矛盾していないか
- 依存関係が正しく保たれているか
- リンクや参照が正しく機能するか

**2. ベストプラクティス**:

- 既存の設計パターンや記述スタイルに従っているか
- 重複や不整合を生んでいないか
- 既存のルールやガイドラインに準拠しているか

**3. セキュリティ**:

- 既存のセキュリティ設定を破壊していないか
- 新たな脆弱性を導入していないか
- `.gitignore` 等の設定が適切に保たれているか

**4. 影響評価**:

- 関連ドキュメントの更新が必要か（README.md, CLAUDE.md, CHANGELOG.md等）
- 他の機能やワークフローへの影響はないか
- 既存のテンプレートやコマンドが正しく動作するか

---

### Phase 3: プロジェクト全体の整合性レビュー

**スコープ**: プロジェクト全体

#### 観点

**1. 方針整合性**:

- プロジェクトの原則や方針に沿っているか（CLAUDE.md 参照）
- ディレクトリ構造やファイル配置が適切か
- プロジェクトの目的や価値観と一致しているか

**2. ベストプラクティス**:

- 命名規則がプロジェクト全体で統一されているか
- ドキュメント体系が一貫しているか
- コーディングスタイルやフォーマットが統一されているか

**3. セキュリティ**:

- プロジェクト全体のセキュリティポリシーに準拠しているか
- プライバシー保護の観点で問題ないか
- 公開リポジトリとして適切か

**4. 拡張性**:

- 将来の拡張に対応できる設計か
- スケーラビリティの問題はないか
- 他の機能との統合が容易か

---

### 最終チェック: マージ前の総合確認

**スコープ**: PR全体とGit状態

#### 観点

**1. 完全性**:

- すべてのレビューフェーズが完了しているか
- 発見された問題がすべて解決されているか
- レビューコメントに未対応事項がないか

**2. 品質保証**:

- PR テンプレートのテストプラン項目がすべて完了しているか
- ドキュメントが最新か（CHANGELOG.md, README.md等）
- コミットメッセージが適切か

**3. Git状態**:

- コミットメッセージが Conventional Commits に従っているか
- ブランチが最新の main から分岐しているか
- コンフリクトがないか
- 不要なファイルがコミットされていないか

**4. マージ可否判断**:

- すべてのチェックが通っているか
- 承認可能か、追加修正が必要か
- マージ後の影響は許容範囲内か

---

## 横断的観点のマトリックス

すべてのフェーズで以下の観点を確認します：

| 観点 | Phase 1 | Phase 2 | Phase 3 |
|------|---------|---------|---------|
| **正確性** | ✅ 変更内容の技術的正確性 | ✅ 既存との整合性 | ✅ プロジェクト方針との整合性 |
| **ベストプラクティス** | ✅ ローカルな規則 | ✅ 既存パターンとの統一 | ✅ プロジェクト全体での統一 |
| **セキュリティ** | ✅ 機密情報チェック | ✅ 既存セキュリティ設定 | ✅ プロジェクトポリシー準拠 |
| **保守性/拡張性** | ✅ コードの保守性 | ✅ 影響範囲の最小化 | ✅ 将来の拡張性 |

## 変更種別ごとの重点観点

### feat（新機能）

- **Phase 1**: 設計の妥当性を重視（新機能の実装が適切か）
- **Phase 2**: 既存機能との統合を重視（既存機能との整合性）
- **Phase 3**: プロジェクト方針との整合性を重視（全体の方向性と一致するか）

### fix（バグ修正）

- **Phase 1**: 根本原因の特定と修正の正確性（バグが確実に修正されているか）
- **Phase 2**: 副作用がないか慎重に確認（修正が他に影響していないか）
- **Phase 3**: 同様のバグが他にないか確認（再発防止）

### docs（ドキュメント）

- **Phase 1**: 内容の正確性と可読性（ドキュメントが正しくわかりやすいか）
- **Phase 2**: 他のドキュメントとの整合性（矛盾や重複がないか）
- **Phase 3**: ドキュメント体系全体での一貫性（全体の構造と整合しているか）

### refactor（リファクタリング）

- **Phase 1**: 動作が変わっていないか（機能の同一性）
- **Phase 2**: 依存関係が保たれているか（既存機能への影響なし）
- **Phase 3**: 設計改善がプロジェクト全体に良い影響を与えているか

### style（スタイル修正）

- **Phase 1**: フォーマットが統一されているか
- **Phase 2**: 既存のスタイルとの一貫性
- **Phase 3**: プロジェクト全体のスタイルガイドラインとの整合性

### test（テスト追加）

- **Phase 1**: テストケースが適切か（網羅性、正確性）
- **Phase 2**: 既存のテストと重複していないか
- **Phase 3**: テスト戦略全体との整合性

### chore（ビルド・ツール変更）

- **Phase 1**: 設定が正しいか
- **Phase 2**: 既存のワークフローを破壊していないか
- **Phase 3**: プロジェクトのビルド・デプロイ戦略との整合性

## レビュー担当

### 開発者

- 最終的な判断と承認
- 必要に応じて詳細なレビュー

### Claude

- セルフレビュー（すべてのフェーズ）
- レビュー結果をPRコメントとして記録
- 発見した問題の修正提案

---

## レビュー結果の記録方法

**【重要】レビューは PR 上で実施し、結果をコメントに記録する**

詳細は [CLAUDE.md のレビュープロセスのルール](../CLAUDE.md#レビュープロセスのルール) を参照してください。

### 記録方法

1. **各フェーズのレビュー結果をPRコメントに記録**:
   - コメントタイトル: `## レビュー: [観点名]`（例: `## レビュー: 変更内容の詳細`）
   - レビュー内容を記載
   - 問題点と改善提案を明記
   - 「問題なし」の場合もその旨をコメント

2. **修正が必要な場合**:
   - 修正方針をPRコメントに記載
   - 修正後、修正内容を説明するコメントを追加
   - 修正コミットのハッシュも記録

3. **レビュー完了時**:
   - すべてのフェーズが完了したら、その旨をコメント
   - 承認可能かどうかを明記
   - 最終チェック結果もコメントとして記録

## ベストプラクティス

### レビューの心構え

1. **建設的であること**: 批判ではなく、改善のための提案を行う
2. **具体的であること**: 「良くない」ではなく「○○の理由で△△に変更すべき」と伝える
3. **優先順位をつける**: 重大な問題と軽微な改善提案を区別する
4. **学びの機会**: レビューを通じて新しい知識や視点を得る

### 効率的なレビュー

1. **段階的にレビュー**: Phase 1 → Phase 2 → Phase 3 の順に確認
2. **チェックリストを活用**: [review-checklist.md](review-checklist.md) を参照
3. **早期フィードバック**: 問題を見つけたら早めに指摘
4. **自動化できる部分は自動化**: linter や formatter を活用
